## プロジェクト概要

BeFake.は「燃えないSNS時代の自己満足」をコンセプトとした日記アプリ。
ユーザーが書いた日記をAIが時間経過とともにポジティブに改竄し、「都合のいい自分史」を作ることで自己肯定感を高める。
他人への共有・比較・承認を排除した、完全に自分だけのプライベートな体験を提供する。

## 機能要件

### FR-01: ユーザー認証・アカウント管理
- Amazon Cognitoを使用した認証基盤
- 認証方式は設計フェーズで決定（メール＋パスワード、Googleソーシャルログイン等を検討）
- ユーザーごとに完全に独立したデータ領域を保証
- アカウント削除時は全データを完全消去

### FR-02: 日記記録（メモ投稿）
- テキスト入力によるメモ投稿
- 音声入力対応（音声→テキスト変換）
- 1日に複数回のメモ投稿が可能（移動中の音声メモ＋夜のテキスト入力など）
- 投稿日時の自動付与
- 箇条書き・自由記述どちらも受け付ける
- 投稿されたメモは「その日の断片」として保存され、日次バッチ処理で1つの日記に統合される

### FR-03: AI日次統合・要約（日次バッチ）
- Amazon EventBridge Schedulerが毎日深夜（例：00:00）にその日の全メモを収集
- Amazon Bedrockがその日の全メモを1つの日記として統合・要約
- 要約結果をその日の「日記」としてDynamoDBに保存（改竄なし・これが起点）
- 使用モデルは設計フェーズで決定（Claude 3 Sonnet / Haiku / Opus を検討）
- 統合・要約ワークフローはAWS Step Functionsで管理
- メモ投稿自体はAPIが即時レスポンスを返す（Bedrock処理はユーザーをブロックしない）

### FR-04: AI改竄（Fake化）スケジュール
- 記録から7日後：改竄プロセス開始
- 記録から14日後：中間改竄（ネガティブ表現を中立〜ポジティブに変換）
- 記録から30日後：最終改竄（「そうだった気がする自分史」として完成）
- Amazon EventBridge Schedulerでスケジュール管理
- AWS Step Functionsで改竄ワークフローを制御
- 改竄の方向性・強度は設計フェーズで決定

### FR-05: 原文保存とプライバシー
- 改竄前の原文はS3に保存（暗号化済み）
- デフォルトでは原文は非表示
- ユーザーが設定から原文を確認できる（オプトイン）
- 原文と改竄後の内容は明確に区別して管理

### FR-06: フィードバック通知
- アプリ内通知で過去の記録（改竄済み）をフィードバック
- 通知頻度は設計フェーズで決定（毎日・週1・月1・ユーザー設定を検討）
- 通知内容：「〇日前、あなたはこんなことがあったようです」
- Amazon EventBridge Schedulerで通知スケジュール管理

### FR-07: 日記閲覧
- 改竄済みの日記一覧表示（タイムライン形式）
- 日付・月・年でのフィルタリング
- 設定から原文の表示/非表示を切り替え可能

### FR-08: プラットフォーム対応
- Webアプリ（Next.js / AWS Amplify）
- ネイティブモバイルアプリ（iOS / Android）
- 両プラットフォームで同一のバックエンドAPIを共有

## 非機能要件

### NFR-01: セキュリティ・プライバシー
- AWS KMSによるカスタマー管理キー（CMK）でのデータ暗号化
- 保存時・転送時の暗号化を徹底
- ユーザーデータは他ユーザーから完全に隔離
- 日記データは絶対に外部共有しない設計
- セキュリティ拡張機能：**無効**（ハッカソン向けPoC）

### NFR-02: パフォーマンス
- テキスト投稿APIのレスポンス：2秒以内（「保存しました」が返るまで）
- 音声投稿：音声ファイルのS3保存完了まで2秒以内（「保存しました」を返す）
  - 文字起こし（Amazon Transcribe）は非同期で実行、完了後にDynamoDBを自動更新
  - 変換中はUI上で「変換中...」と表示
- AI統合・要約処理：非同期バッチ（ユーザーをブロックしない、深夜に自動実行）
- AI改竄処理：バックグラウンド非同期処理

### NFR-03: スケーラビリティ
- サーバーレスアーキテクチャによる自動スケール
- DynamoDBのオンデマンドキャパシティ
- Lambda関数の自動スケーリング

### NFR-04: 可用性・監視
- Amazon CloudWatchによるログ・メトリクス監視
- Lambda・Step Functionsのエラーアラート設定

### NFR-05: テスト方針
- プロパティベーステスト（PBT）：**部分的適用**
  - 純粋関数（AI改竄ロジック・日付計算）に適用
  - シリアライゼーションのラウンドトリップに適用

### NFR-06: インフラ管理
- AWS CDK（TypeScript）によるフルIaC
- 全インフラをコードで管理・再現可能

## 設計フェーズで決定する事項

以下の項目は要件分析時点では「後で決定」とされており、設計フェーズで確定する：

1. **認証方式の詳細**：メール＋パスワード、Googleソーシャルログイン等の組み合わせ
2. **AI改竄の方向性・強度**：軽微〜強め、またはユーザー設定可能
3. **フィードバック頻度**：毎日・週1・月1・ユーザー設定
4. **Bedrockモデル選択**：Claude 3 Sonnet / Haiku / Opus
5. **モバイルアプリの技術詳細**：React Native / Expo の構成
